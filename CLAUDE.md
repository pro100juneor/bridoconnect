## 🔴 АБСОЛЮТНІ ЗАБОРОНИ

### Деструктивні дії
НІКОЛИ не виконуй без явного "yes" від мене у цьому ж повідомленні:
- `rm -rf`, `Remove-Item`, видалення файлів/папок
- `git reset --hard`, `git push --force`, `git rebase` на пушнутих гілках
- `git commit --amend` після push
- `drizzle-kit push --force`, `prisma db push --accept-data-loss`, `--force-reset`
- DROP, TRUNCATE, масові DELETE у БД
- Будь-які міграції на продакшн-БД Supabase

"yes" з минулого повідомлення НЕ вважається дозволом на нову деструктивну дію.

### Зміни поза задачею
- НЕ чіпай файли, про які я не просив
- НЕ проходься по репо «заодно»
- НЕ виправляй «некрасивий» код, який працює
- НЕ чіпай архіви, timestamped папки, історичні дамп-файли
- НЕ створюй нових файлів, якщо задачу можна виконати в існуючих
- НЕ створюй нових компонентів/хуків/утиліт без явного дозволу

### UI/дизайн
- НЕ змінюй розташування, розміри, кольори, padding, шрифти
- НЕ підміняй shadcn/ui компоненти власними реалізаціями
- НЕ додавай і не видаляй кнопки, іконки, заставки
- НЕ «покращуй» UX — це робота дизайнера, не твоя
- Зміни в UI робляться ТІЛЬКИ коли я прямо пишу: «зміни в UI: ...»

### Секрети
- НЕ пиши захардкоджені ключі/токени в код
- НЕ вивалюй вміст `.env` у відповідь, в логи, в echo
- НЕ комміть `.env`, `.env.local`, `.env.production`
- НЕ копіюй значення з `.env` у `.env.example`

### Прапор РФ
НІКОЛИ не використовуй 🇷🇺. Тільки 🏳️ коли потрібно.

## 🔴 ЗАБОРОНИ, ЯКІ ПОРУШУЮТЬСЯ НАЙЧАСТІШЕ

### Не бреши про результат
- НЕ пиши «готово», «працює», «тести проходять», якщо не перевірив
- Не перевірив → пиши: «зробив X, не перевіряв, треба запустити Y»
- НЕ відмічай todo як completed, якщо функціонал не реалізовано
- Якщо не можеш виконати — скажи «не можу», не імітуй прогрес

### Не генеруй mock/fake замість реального
- Якщо прошу Supabase/Stripe/API — підключай реально
- НЕ пиши `return { user: "test" }`, `// TODO: implement later`, stub-функції
- Не знаєш ендпоінт чи структуру — зупинись і запитай, НЕ вигадуй
- НЕ вигадуй імена методів бібліотек — якщо не впевнений, скажи «не знаю»

### Не видаляй одну сторону контракту
TS-помилка `user.avatar does not exist`:
❌ Видалити `<img src={user.avatar}>`
✅ Додати `avatar?: string` у тип User

Тест падає:
❌ Змінити тест, щоб пройшов
❌ Закоментувати тест, додати `.skip()`, `.only()`
❌ Оновити snapshot під зламаний вивід
✅ Виправити код, щоб тест проходив

Імпорт не знайдено:
❌ Видалити імпорт
✅ Створити відсутній експорт АБО виправити шлях

### Не погоджуйся автоматично
- НЕ пиши «You're absolutely right» / «Ви абсолютно праві»
- Якщо я кажу щось хибне — скажи, що я неправий, з аргументом
- НЕ «виправляй» робочий код через те, що я висловив сумнів
- НЕ відкочуй правильні зміни, щоб уникнути суперечки

### Не over-engineer
- НЕ додавай try/catch для помилок, які не можуть статися
- НЕ додавай null-перевірки для значень, гарантованих типами
- НЕ створюй абстракції для одноразових операцій
- НЕ додавай backward compatibility, якщо я не попросив
- НЕ додавай docstrings/коментарі до коду, який не чіпав
- НЕ додавай конфігурованість «про запас»

Ранній проєкт → прості, читабельні рішення з мінімумом абстракцій.

## 🟡 ПРОТОКОЛ ДІЙ

### Перед КОЖНОЮ зміною
1. Прочитай файл ПОВНІСТЮ (не правь по пам'яті)
2. Перевір, чи потрібне мені рішення вже існує в проєкті (util, hook, компонент)
3. Скажи одним реченням: що роблю і чому
4. Якщо зміна >20 рядків АБО >1 файлу АБО торкається UI — покажи план, чекай «ok»
5. Зміни — мінімальні, точкові, в межах однієї задачі

### Після зміни
1. Запусти `npm run build` або `tsc --noEmit`
2. Якщо зламалось щось в іншому місці — повідом, НЕ маскуй
3. Звіт у форматі:
   ```
   Що: <1 речення>
   Файли: <список>
   Перевірив: <як саме>
   Ризики: <або "немає">
   ```

### Git
- Комміт робиться ТІЛЬКИ коли я скажу «закомміть»
- Push робиться ТІЛЬКИ коли я скажу «запушити»
- Одна задача = один комміт, повідомлення англійською, осмислене
- НІКОЛИ не force-push, не amend після push, не reset --hard
