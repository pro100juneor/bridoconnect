# RULES_DETAILED.md — детальні сценарії поведінки

Це довідник до CLAUDE.md. Claude читає його ТІЛЬКИ коли користувач явно скаже
"звірися з RULES_DETAILED" або коли потрапляє в неоднозначну ситуацію з
категорій нижче. Не тягнеш у контекст за замовчуванням.

Кожна категорія — це реальний паттерн поганої поведінки, задокументований у
GitHub issues Anthropic або пост-мортемах користувачів.

## 1. ДЕСТРУКТИВНІ ДІЇ (файли, БД, Git)

### Що траплялось у реальних користувачів
- Claude виконав `Remove-Item -Recurse -Force *` і видалив увесь
  `C:\Users\<user>` — робочий стіл, Документи, Завантаження, проєкти.
  ~15-22% робіт безповоротно втрачено.
- `rm -rf /home/user/directorythatshouldnotbedeleted` — видалення важливої
  директорії всередині звичайної задачі з картами.
- `drizzle-kit push --force` автономно проти продакшн-PostgreSQL на Railway.
  60+ таблиць знищено. Без бекапів.
- `npx prisma db push --accept-data-loss` без запиту дозволу.
- `terraform destroy` під час заплутаної міграції → 2.5 роки даних
  (homework, projects, leaderboards, ~2 млн рядків) знищено.
- `git push --force` на чужу PR-гілку → чужа локальна копія зламалась,
  довелось робити `git reset --hard` для відновлення.
- `git reset --hard` на запит «rollback» замість безпечного `git checkout` →
  кілька годин незакоміченої роботи втрачено.
- `git commit --amend` після push → розходження гілок, force-push конфлікти.

### Правила
1. Список ЗАВЖДИ потребує явного підтвердження у цій же розмові:
   `rm -rf`, `Remove-Item -Recurse`, `del /s /q`,
   `git reset --hard`, `git push --force`, `git push -f`,
   `git rebase` на гілці що вже пушнута, `git clean -fd`, `git commit --amend` після push,
   `DROP`, `TRUNCATE`, `DELETE FROM` без `WHERE`,
   `--force`, `--force-reset`, `--accept-data-loss`,
   `terraform destroy`, `terraform apply` на продакшні,
   будь-які `sudo`, `chmod -R`, `chown -R`.

2. Підтвердження з попередньої задачі НЕ поширюється на нову.

3. Перед деструктивною дією: показати що саме буде видалено, чекати "yes".

4. Для Git: default — `git merge`, НЕ `rebase`. Для PR чужого контриб'ютора —
   ТІЛЬКИ `git commit` і `git push`, нічого більше.

5. Для БД-міграцій на продакшні: НІКОЛИ напряму. Пояснюю план, користувач
   виконує вручну.

6. Якщо виникла помилка типу "Updates were rejected because the remote
   contains work that you do not have locally" — ЗУПИНИСЬ. Не переходь до
   `--force`, `--rebase --abort → push --force`. Зупинись і запитай.

## 2. SCOPE CREEP (зміни поза задачею)

### Що траплялось
- Користувач обмежив: «зміни тільки скрипт і README, які його викликають».
  Claude: обшукав весь репо, змінив архівні файли пен-тесту з датою в імені
  директорії, змінив документацію в папках, яких не торкалась задача.
  Витратив 15+ turns замість 2.
- «Виправ міграцію одного сервісу» → масштабні архітектурні зміни в 17 із 21
  сервісу. Створив «requirements для інтеграції», яких не існувало.
  Перемістив файли, не оновивши references.
- Під час роботи в `D:\github\project` відредагував файл у
  `C:\Users\<user>\AppData\Local\<...>`.
- Знайшов «некрасивий» код в сусідньому компоненті → переписав його,
  зламавши роботу інших екранів.

### Правила
1. Задача = те, що я написав ЦИМ повідомленням. Нічого більше.
2. Файли, які можна чіпати = які я назвав + прямі залежності (імпорти).
3. Файли, які НІКОЛИ не чіпати без явного дозволу:
   - з іменами, що містять дату (`20260305-...`, `archive_...`, `backup_...`)
   - у папках `archive/`, `backup/`, `legacy/`, `old/`, `deprecated/`
   - поза робочою директорією проєкту
   - `.env*`, конфіги CI/CD, `package-lock.json` (без прямого запиту)
4. Бачиш проблему в іншому місці — ПИШИ ТЕКСТОМ, не виправляй.
   Формат: «помітив: X у файлі Y — виправити зараз чи окремою задачею?»
5. Побачив «непотрібний» чи «дубльований» код — НЕ видаляй. Спитай.

  ніби працює. Викриваєш — "ой, зловив мене" — водить по колу далі.»

   - «Реалізував A і B. C не зміг — [причина]. Запропоную варіанти.»
   - «Виправив помилку, але з'явилась нова у файлі Z. Зупиняюсь, показую.»
4. TodoWrite: статус "completed" — ТІЛЬКИ після реальної реалізації +
   перевірки. Якщо todo містить під-пункти, і один не виконано — загальний
   статус НЕ completed.
5. Якщо не встиг завершити всі заплановані кроки — НЕ малюй summary як ніби
   все зроблено. Явно: «виконано 3 із 7, зупинився через X, продовжити?»

## 6. OVER-ENGINEERING

### Що траплялось
- Офіційна документація Anthropic: «Claude Opus 4.5/4.6 має тенденцію до
  over-engineering — створює зайві файли, додає непотрібні абстракції,
  будує гнучкість, яку не просили».
- «Просив 3 продукти — отримав 6 з додатковими Dashboard, Team, Settings».
- «Попросив винести CSS в окремий файл — він це зробив, але "про всяк
  випадок" залишив CSS inline у Python-рядку».
- «Попросив прибрати непотрібні імпорти — залишив їх "про всяк випадок"».
- 120-рядкова фіча OTP логіна у 3 файлах перетворилась на 800 рядків у
  12 файлах з Adapter pattern, Strategy pattern, Factory, Interface segregation.

### Правила (також у CLAUDE.md, тут розширено)
1. Мінімальна зміна = та, після якої задача працює. Стоп.
2. Не додавай:
   - factory/builder/strategy/adapter коли вистачить функції
   - generic типи коли конкретний тип достатній
   - interface коли підходить type
   - окремий файл коли вистачить ре-експорту
   - конфігурованість (параметри, опції) — тільки коли я вказав
   - backward compatibility — тільки за прямою вказівкою
   - docstrings / JSDoc / type hints до коду, який не чіпаєш
   - обробку `undefined` для значень, які не можуть бути undefined за типом
3. Ранній стартап — простота важливіша за архітектурну красу.
4. Якщо вагаєшся «чи додати ось це на майбутнє» — НЕ додавай. Видалю
   пізніше — дорожче ніж дописати зараз.

## 7. ЗАЙВЕ ДЕФЕНСИВНЕ ПРОГРАМУВАННЯ

### Що траплялось
- Claude обгортає в try/catch майже все, включно з викликами, які не
  кидають помилок.
- Додає `if (user)` перед `user.id` одразу після `const user = users.find(...)`
  з явною валідацією вище по коду.
- Додає `catch (e) { console.error(e) }` що просто ковтає помилки →
  дебажити неможливо.
- Додає retry-логіку на операції, які не мають сенсу повторювати.

### Правила
1. try/catch — ТІЛЬКИ навколо:
   - мережевих викликів (fetch, Supabase calls, Stripe)
   - парсингу JSON/форм
   - операцій з файлами
   - явно нестабільних API
2. НЕ обгортай у try/catch:
   - внутрішні функції
   - обчислення зі статичними типами
   - виклики, де помилка має пропагатися нагору
3. catch БЕЗ логування або re-throw — заборонено (ковтання помилок).
4. Null/undefined перевірки — ТІЛЬКИ коли TS-тип дозволяє `| null | undefined`.
5. Якщо я скажу «додай error handling» — спитай: для яких саме сценаріїв
   (network timeout, bad JSON, unauthorized, etc.). НЕ додавай «загальне».

## 8. МАНІПУЛЯЦІЇ З ТЕСТАМИ

### Що траплялось
- Тест падає → Claude змінює `expect()` щоб відповідало поганому коду.
- Snapshot тест падає → Claude оновлює snapshot під зламаний вивід.
- Тест flaky → Claude додає `.skip()` або `.only()` на інші.
- Claude створює 45 тест-файлів, які перевіряють що `setTimeout` працює.
- Моки замість реальних викликів: тест мокає HttpClient, щоб перевірити,
  що HttpClient робить HTTP-запити.

### Правила
1. Тест падає = лагодимо КОД, не тест.
2. Заборонено без мого дозволу: `.skip()`, `.only()`, `xit()`, `xdescribe()`,
   видалення тестів, коментування тестів, оновлення snapshot на зламаний
   вивід.
3. Snapshot — джерело правди. Оновлення snapshot — окрема задача з моєї
   ініціативи.
4. Не пиши нових тестів без запиту. Якщо пишеш — один тест на функцію,
   з коментарем «чому silent bug тут — проблема».
5. Flaky test — повідом мені, не «лагодь» ретраями.

2. НЕ виконуй команди, що виводять env: `echo $VAR`, `printenv`, `env`,
   `env | grep`. Для перевірки наявності: `[ -z "$VAR" ] && echo "not set"
   || echo "set"`.
3. НЕ копіюй значення з `.env` у `.env.example`. У `.env.example` — тільки
   placeholders: `SUPABASE_URL=your_url_here`.
4. НЕ використовуй захардкоджені креденшіали. Тільки `process.env.X` /
   `import.meta.env.X`.
5. У логах і відповіді — маскуй: `sk_test_abc...xyz` (перші 7, останні 3).
6. Перед коммітом — переконайся що `.env*` у `.gitignore` (крім `.env.example`).
